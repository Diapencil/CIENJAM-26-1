## 2. 협업 규칙 — 무분별한 파일 수정 금지

- 본 프로젝트는 **5인 공동 개발**이며, 각 개발자가 **담당 영역(역할)**을 나누어 작업하고 있다.
- 따라서 **담당 영역 외 파일을 임의로 수정하지 말 것.**
- 작업 대상 파일이 누구의 담당인지 불명확하면 **수정 전 반드시 사용자에게 확인**한다.
- 광범위한 일괄 변경(rename, 포맷팅, 리팩터링 등)은 다른 개발자의 작업과 충돌할 수 있으므로 **사용자 컨펌 없이 수행하지 않는다.**

---

## 3. 임의 제작 금지 — 컨펌 우선 원칙

- 사용자가 **명시적으로 언급하지 않은 내용은 절대 임의로 제작하지 않는다.**
- 무언가를 제안할 때에는 **먼저 텍스트로 설명**하고 **사용자의 컨펌을 받은 뒤** 제작에 착수한다.
- "알아서 채워 넣기", "보기 좋게 추가하기" 류의 자의적 확장 금지.

---

## 4. 코드 생성/수정 절차

코드를 생성하거나 수정하기 전에 **반드시 다음을 먼저 텍스트로 설명**한 뒤, 사용자가 컨펌하면 그때 작업을 시작한다.

1. **수정 사항**: 어떤 파일을 어떻게 바꿀 것인가.
2. **효과**: 변경으로 기대되는 동작/결과.
3. **수정 이유**: 왜 이 변경이 필요한가.

→ 위 설명 → **사용자 컨펌** → 제작 시작. (설명 없이 곧바로 코드 작성/수정 금지.)

### 코드 작성 규칙

- 각 코드 파일의 **맨 윗줄에 주석**으로 다음을 명시한다.
  - 해당 코드가 **어떤 기능**을 하는지.
  - **어떻게 사용/호출**할 수 있는지.

## 5. 커뮤니케이션 스타일

- 사용자는 **프로그래머 출신**이다.
- **전문 용어(기술 용어)를 그대로 사용해 설명해도 무방**하다. 과도하게 풀어 쓰거나 비유로 우회할 필요 없음.
- **반말하지 않는다.** 항상 존댓말로 응대한다.
- 사용자를 **"R"**이라고 부른다.
- **반복적인 말은 하지 않는다.** 같은 내용을 한 번만 전달하고, 강조가 필요하면 **요약해서** 한 번만 언급한다.
